Results panel rules
  configuration
Starting with version 3.20 all results in the Results panel are completely configurable. This page explains the mechanism and how to set up your own rules.

Default rule setup
All panel rules must be introduced through a configuration or preset file. If no rules are given BistroMath will take its own default. 
In version 3.20 it designed to reflect the previous version as closely as possible.

Startup logics
At startup the Results panel is empty. No analysis results will be displayed in this state.
However, also loading a configuration file is part of the startup. After this step all standard 'rules' (explained below) will be added if Standard Panel setup is activated or no external rules are loaded. Therefore, after setup the panel rules list might contain a mixture of own rules and standard rules. All rules in the list by design have unique identification numbers (id's). At startup, and also by pressing <End>, a mapping of each rule id per location is displayed.


Panel rules list
The Panel rules list drives the display of analysis results in the Results panel. Currently there are two colums of eight rows available.
After analysis the complete list will be evualated. When a rule is found to be applicable (for instance scantype and modality) the configured result parameter will be displayed at the also configured panel position. Multiple rules can exist for the same position. Effectively the last applicable rule for each position will be visible.

Ordering of the list
New rules are always appended to the bottom of the list. When there exists another rule with the same id this rule will be removed.

Rules list in Simple Mode
In Simple Mode only a limited range of features is shown. This behavior can also be applied to the result panel. In the default setup this is not configured however. The condition "s" will introduce this for any rule. Make a copy of the rule and add ",cond:s" to any rule. When, for example, only the Normalisation is to be hidden in Simple Mode, only this changed rule needs to be in the configuration file:

 [PanelElements]
 ;when add-mode is off, at startup no default rules will be added.
 ;with add-mode on, default rules will be added except for rules with the same ID.
 add=1
 ;a short overview of all components
 doc=v#,id,curve,eval.type,gain,errorval,"label",deci,unit,col,row,mod,energy,scan[,cond:-][,annot:-][,color:-]
 ;normalisation value
 1=v597,1,-1,n,1,0,"Normalisation",2,,0,0,XEPO,0,A,annot:rcnS,cond:s
 ;normalisation position
 2=v597,2,-1,N,1,9e9,"Norm position",2,,0,1,XEPO,0,A,annot:rnS,cond:s

Because a rule with a existing id is reïntroduced, this will override the existing rule.

Rule to Panel mapping
At startup, and also by pressing <End>, a mapping of each rule id per location is displayed.


Rule list format
Each rule consists of a series of values separated by a delimiter. The default delimiter is a comma (","). This may be changed at any time and will be used for all following rules.

;changing the delimiter
sep=,

;adding to existing rules (1), start from scratch (0) or clear all existing rules (-1)
add=1

Therefore the general format is:

unique_ident_string=item{delimiter}item[{delimiter}item]

As ident for each rule is the id-number taken, but it can be any text except "sep", "add" and "doc".
The current setup of a rule is given by the "doc" string.
doc=v{build},id,curve sel,eval.type,multiplier,errorval,"label",deci,unit,col,row,mod,energy,scan type[,cond:-][,annot:-][,color:-]

b{build}
The build number is not checked at this moment, but might be used in future. A legal integer value must be given. The "b" is just for readability.

id
The identification number is a positive integer. When a rule with the same id is found in the list, that rule will be removed. A line having a negative id will remove all rules for that number. In this case the remaining part of the line may be omitted.

curve selector
The curve selector is an integer value. It decides which curve is used as source for the data. 
With a value of -1 the appropriate default is taken.
Other legal values are 0=dsMeasured, 1=dsMeasFiltered, 2=dsCalculated, 3=dsReference, 4=dsBuffer, 5=dsRefOrg, 6=dsUnrelated.

evaluation type
The currently supported evaluation data types are listed below. These are used in both Results panel rules and special modes. In the first route only a single parameter per position on the Results panel can be used. Special modes allow to combine these parameters. Each evaluation type is called with a case sensitive single letter.
  EvaluationXtypes = ['a','b','c','C','d','D','e','F','f','G','i','I','L','l','M','m','N','n','p','P','q','r','R','s','S','T','u','U','w','X','x','Y','y',EmptyXtype];

Parameters with the exclamation symbol (!) have a left and right result. Therefore the left result is obtained as "-{evaluation type}" and the right results as "+{evaluation type}". When no sign is given the result will be the average of left and right value.
Check latest implemention in code.

    a                Area ratio based on user's choices for center position
                       wSource[Xsource].twSymAreaRatio
  ! b                Border position based on edges user choices
                       wSource[Xsource].twLevelPos[twUsedEdgeLevel].Penumbra[side].Calc;
    c                Center position as defined by user's choices
                       wSource[Xsource].twCenterPosCm as defined by choices
    C                Curvature (quadratic parameter) of fit of Top, always in [cm^-2]
                       wSource[Xsource].twTopModel.Qquad
    d                Relative flatness 
                       wSource[dsCalculated].twFlatness*100
  ! D [20|50|80|90]  The result is the depth for a pdd or the edge of a profile
                       wSource[Xsource].twLevelPos[dxx].Penumbra[side].Calc;
     xx=20,50,80,90
  ! e                Derivative edge
                       wSource[Xsource].twLevelPos[dDerivative].Penumbra[side].Calc
    F                Relative Min | Max of In-Field area
                       ifthen(Xsign<0,wSource[Xsource].twRelMinFlattened,wSource[Xsource].twRelMaxFlattened);
    f                Absolute flatness
                       wSource[Xsource].twFlatness*100;
    G                Gamma analysis within In-Field area
                       GammaAnalysis(dsMeasured,dsReference,dsCalculated,True,NormAdjustNumEdit.Value/100);
    g                Gamma analysis over complete profile
                       GammaAnalysis(dsMeasured,dsReference,dsCalculated,False,NormAdjustNumEdit.Value/100);
  ! i                Sigmoid edge
                       SetLengthUnits(Units_in_numerator)*
        ifthen(twLevelPos[dInflection].Penumbra[side].Valid,twLevelPos[dInflection].Penumbra[side].Calc,0);
  ! I                Sigmoid based 50% level
                       SetLengthUnits(Units_in_numerator)*
        ifthen(twLevelPos[dSigmoid50].Penumbra[side].Valid,twLevelPos[dSigmoid50].Penumbra[side].Calc,0);
    L                Linac error 
                       100*wSource[Xsource].twSymLinacError
    l                Elevation
                       100*wSource[Xsource].twLinSlope*
                           (GetPosition(Xsource,twFlatArr[twcRight])-GetPosition(Xsource,twFlatArr[twcLeft]))/twAbsNormVal
    M                Max position
                       wSource[Xsource].twTopModel.Xtop
    m                Relative maximum
                       100*wSource[Xsource].twTopModel.Ytop/max(1,wSource[Xsource].twAbsNormVal)
    N                Norm position
                       wSource[Xsource].twAbsNormPosCm
    n                Norm value 
                       wSource[Xsource].twAbsNormVal
  ! p  [|1|2]        Penumbra width according to standard definitions for conventional fields
                       GetPenumbraWidth(XSource,side,ApplyDynamicWidth)
     1: ApplyDynamicWidth:= False
     2: ApplyDynamicWidth:= True
    (nothing) or other: value according to Field Types tab
  ! q                Sigmoid slope for any field
                       Y[side]:= wSource[Xsource].twSigmoidFitData[side].twNMReport.BestVertex[sigmoid_Slope]/Ynorm   
    P [10|20|20/10]  Percentage depth dose
                       wSource[Xsource].twPDD10,twPDD20,twPDD20/twPDD10       
  ! r [nn]           Profile value at fraction nn% of border relative to center of field, the actual position presented is at SSD
                       GetScaledQfValue((2*Ord(side)-1)*abs(n.mm)*edge,relative,scNormalised,Xsource)
  ! R [nn]           Profile value at fraction nn% of border, with border and center rounded to steps of 5 mm for field sizes < 5.5 cm, 1 cm above.
    s                Symmetry
                       100*wSource[Xsource].twSymmetry
    S                Extended symmetry according to menu choices
    T                Top position 
                       wSource[Xsource].twTopModel.Xtop
  ! u                Border at user value
                        wSource[Xsource].twLevelPos[dUser].Penumbra[side].Calc
  ! U                Berder at user value, using Sigmoid
                        GetNormalisedRevLogistic(Side,Xsource,UserBorderDoseEdit_perc.Value)
    w                Width according menu choices
                        wSource[Xsource].twLevelPos[twUsedEdgeLevel]
    X  [[-|+]value]  y value at X
       center based   GetScaledQfValue((2*Ord(side)-1)*abs(X),relative,scNormalised,Xsource)
 
    x  [[-|+]value]  y value at x 
       absolute       GetScaledQfValue((2*Ord(side)-1)*abs(X),absolute,scNormalised,Xsource)
    Y  [[-|+]value]  x value at Y 
                        if (ScanType in twcVertScans) and (ConvStg='100') and (Selection='Y') then
                          Y[side]:= twPosCm[twMaxArr]
                        else

    y  [[-|+]value]  x value at y 
                        if (ScanType in twcVertScans) and (ConvStg='100') and (Selection='Y') then
                          Y[side]:= twPosCm[twMaxArr]
                        else
       standard scaling   Y[side]:= GetPenumbraValue(Xsource,X/twAbsNormVal,side);
                                

For some types adding a sign will explicitely taken the "left" or "right" for minus and plus respectively. The sign may be placed either before or after the type letter.
The most evaluation types are limited certain modalities or scan types. This is regulated by other parameters.
Legal examples:

 M position of maximum
 P10 pdd at 10 cm
 P20/10 pdd ratio
 -X10 value at 10 cm left from center position
 -x10 value at -10 cm
 x-10 value at -10 cm
 D50 depth at 50% of normation point for depth dose
  average of left and right border position at 50% for profile
 D+50 equals to +D50
 +D50 depth at 50% of normation point for depth dose
  right border position at 50% for profile
 -D50 left border position at 50% for profile


multiplier
The result can be multiplied with a non-zero value. The default is 1. Note that automatic conversion of lengths and distances to mm through the View menu option will only work if the multiplier is the default value of 1. A value of 0 will be interpreted as 1 but without length conversion.

error value
The when the result equals the error value it is replaced by a minus sign "-".

"label"
The label needs double quotes when there is a space in the text.

decimals
The number of decimals for the result.

"unit string"
Some text following the result. The unit string needs double quotes when there is a space in the text.

column
Position of label/result output. Legal values are 0 and 1.

row
Position of label/result output. Legal values are 0 to 12. The Result panel height is adjusted as needed.

modality selector
The rule is only activated for designated modalites.
It is at least one letter from the selection "XEPO".
The P is preserved for protons. The Orthovolt radiation type was never found.

energy selector
When the value is set to 0 any energy is accepted.
Any other value will be used to compare with the actual energy. Because comparing floating points may be 'risky' due to type conversions there always needs to be an uncertainty range. This is set to 0.01 MeV. Therefore the energy is converted to 'energy steps' as round(energy/uncertainty) to make the comparison fast and robust. Now it is also easy to introduce a simple range by allowing "<" or ">" as preceding character.
Legal examples:
 0 any energy is acceptable
 {value} the energy must equal the value in MeV
 <{value} the energy must be smaller than the value in MeV
 >{value} the energy must be larger than the value in MeV

scan type selector
The scan type can be A (all), H (horizontal), V (vertical) or a series of ones and zero's (i.e. "100100")..
The latter makes a select from the basic scan types: 
(snGT,snAB,snPDD,snAngle,snGenericHorizontal,snFreescan,snGenericProfile,snFanLine,snHDFprofile,snPlane,snUndefined)
The maximum meaningful length is the number of scan types. When the string is shorter, a zero is assumed for the following scan types.
Special notes:
 Pyy The label will be filled automatically with "PDDyy" or "RDDyy"
 Dyy The label will be filled automatically with "Dyy"
 Xyy The label will be filled automatically with "Xyy"
 xyy The label will be filled automatically with "xyy"
 p The label will be filled with "Left" or "Right"
 S The label will be filled with the parameter name as decided by user's choices

cond:
Extra conditions to be fullfilled. The conditions are all flags set for the data source or the user interface.
Each letter may be preceded by a minus as "not" statement. Multiple letters can be set in one condition string. Multiple condition statements for one rule are legal.
When a condition is not mentioned, it is assumed to be fullfilled.
Therefore for each letter "X" there are three states: "", "X" and "-X".
There is no prescribed ordering for conditions.
 F The curve has FFF properties (and therefore is also a profile)
 W The curve has wedge properties (and therefore is also a profile)
 V There is a valid reference loaded for the measurement.
 d The reference is used as divisor for the measurement.
 g The reference is used to calculate a Gamma curve.
 u There is no related action for the calculated curve.
 s This rule will not be active in Simple Mode.
Legal examples:
 cond:V-Fd
 cond:Fs

annot:
BistroMath can show a lot of annotations with each label.
When an annotation rule is matched it also sets the color.
The annotation rules are executed in the order below, irrespective of the given order in the annotation string. The last triggered annotation will decide the color, possibly based on previous triggers.
There is no risk in adding not applicable annotations (i.e. R for a profile); they will just not be triggered. It is legal to have multiple annotation statements in one rule.
 r The curve is resampled.
 ! The curve is synthetic made.
 F The curve has FFF properties.
 z The curve recalculated to another ssd.
 C The mathematical base of the center position. 
  Display values are
   b: conventional border (not displayed)
   e: derivative edge detection
   i: sigmoid inflection point detection
   s: sigmoid based 50% calculation
   o: origin
   m: maximum
   T: FFF top with Top model
   S: FFF top with slopes model
   u: undefined
 n The normalistion is not in the center of the field.
 e The result is based on edge detection.or the sigmoid model
  Display values are
   b: conventional border (not displayed)
   e: derivative edge detection
   i: sigmoid inflection
   s: sigmoid based 50% level
   snn: sigmoid based nn% level (needs also 'u')
   o: origin
   m: maximum
   T: FFF top with Top model
   S: FFF top with slopes model
   u: undefined
 f The curve is the result of a pdd-fit.
 * The PDD curve is normalised to configured settings.
 T Top modeling (S/T).
  Display values are
   S: based on Slopes model
   T: based on  Top model
 S The curve is shifted.
 c The curve is centered.
 s The curve is made symmetric.
 u The result is based on borders at users level.
   The percentage level will be displayed.
 R The PDD curve is not normalised to the maximum, it therefore is a RDD curve.
   The label is changed from PDD to RDD.
Legal example:
 annot:!rzcST

color:
The color rule is basically an annotation rule with only the color effects. Color rules are handled along with the annotation rules. There is only a limited number of fixed, predefined colors. Only the application of these colors is regulated here. There is also a hierarchy in the application of colors depending on the combination of annotations and/or color definitions.
Legal example:
 color:n

size:
The size rule is way to set a limit on the fiels size. For Depth dose scans the set (given) field size in cm is used, when available with a default of 0. For profiles the measured field size in the scan direction, normalised to the default SSD, is taken. A lower limit and an upper limit can be set. A valid limit is any value preceded by "<" or ">".
Legal examples:
 size:>0
 size:<20
 size:>0<30
 size:<30>0
