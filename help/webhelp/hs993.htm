History v3.00 - v3.25

version 3.25, 23/11/2018
build 641
  • X/Y/Z/dose data lines without any known header are not yet supported but did introduce problems. For now these data sets are safely ignored. Implementation is put on the wish list.
  • A new Generic to Electron item is introduced in the measurment menu.
  • The ill-chosen name "linearity" as alternative measure for symmetry is now replaced with "elevation". No further substantial changes.
  • For the Gamma Analysis the default number of search steps per mm is increased to 5. The install will also set this value.
  • Bug repair: in (initialisation part of) match procedure could in seldom cases go in to endless loop.
  • Introduction of Smart scale electron PDD option in View menu.

    version 3.24, 30/09/2018
    build 633
  • A bug was found in the Gamma Analysis which could lead to out of data range operations.
  • Now the last directory is kept when storing a file.
  • When the position of reference profile is matched to the measured profile the normalisation position and its strategy on whicht is is derived are taken from measured profile.

    version 3.23, 27/09/2018
    build 632
  • A measurement renormalisation mode is introduced.
  • Bugfix: In the histogram for a Gamma Analysis were also invalid points (with a value of zero) included. This influenced the confidence limit, based on the histogram. Also, now both the histogram based value and the "raw data" value are shown. The limitation to the In-Field area is clarified in the help item.
  • The default display rule for the confidence limit is now expanded to alle scan types.
  • The file save dialog path was unintentionally reset to the startup directory. Now it keeps the chosen path.
  • SpecialMode-strings can now (optionally) be placed into a separate [SpecialMode] section of the ini-file. Previous versions did not preserve these values at all. SpecialMode-strings in the [Analyseform] section still will not be preserved. Note that comments will not be preserved.

    version 3.22, 14/09/2018
    build 620
  • Bugfix: Edge detection fallback failed to trigger during analysis because underlying data were made invalid during filtering in derivative calculation. The underlying code was introduced in version 3.21. This filter behavior is switched off now for the derivative calculation.
  • Bugfix: The interpretation of the scanangle is changed: the diagonal TAGB is swapped with TBGA. Also this bug was introduced during code revisions in version 3.21.
    build 628, 11/10/2018
  • Show units of horizontal axis (cm or mm).
  • Bugfix: Sometimes analysis failed due to out of boundary error in left-over non-functional code. Code removed.
  • Number of decimals (in default setup) for PDD20/10 reduced from 4 to 3.
  • Assumptions of being a depth dose or a profile for poorly defined data are now explicitely stated in the messages tab.
  • Number of decimals for penumbra width in default panel setup increased to 2.
  • The speed of the sigmoid fit was severely limited by the reporting on the messagebar. Now the reporting is limited to loglevel 2 and higher for the measurement data only.
  • Minor display bug: the error points for a pdd fit were always displayed on a cm scale.
  • "Add" mode introduced in the Panel setup. Also, introducing a rule with a negative ID will remove any rule with that number.
  • Minor bugfix: Ability for comparison of unrelated profiles was lost.

    version 3.21, 03/06/2018
    build 611
  • To avoid confusion for a pdd with a non-maximum normalisation also the values at the cursor position are annotated.
  • New feature: the possibility to assign the origin value from the StarCheck 2D-array GT line to the other lines as extra point.
  • Limit to In-Field area option made available for Gamma Analysis.
  • Avoid needless sigmoid fits.
  • Minor bug: merge settings were hidden behind other groupbox in Settings tab.
    build 611, 11/06/2018
  • New option to normalise the source data in the Signal correction sub menu.
  • Bug found in presented values after shift in a depth dose. Related to not updated version of filtered version of the measurement.
  • Display error: "RDD" and "PDD" were swapped.
    build 615, 13/07/2018
  • Improved implementation for SpecialMode2.
    build 616, 23/07/2018
  • The alias list is now also applied to the measurement device name. This opens new possibilities for multiple devices with a shared reference but different name. This is the case the StarcheckMaxi.
    build 617, 21/08/2018
  • Bug found in scan angle definition for some diagonal scans due to wrong ordering of instructions in code in version 3.09 after an attempt to cover also 90 degrees scan angles. In these cases the scan angle was set to 90 degrees for these scans. This should be 45 or 135 degrees for true diagonal scans. Also, BistroMath now relies on its own calculation of the scanangle and overrides any scan angle information in a data file. Due to this switch in strategy there might be a mixup in the reference files: possibly files with '045' have to be renamed to 135 and vice versa.
    build 618, 04/09/2018
  • Too many decimals given for values smaller than 0.1. This might lead to too long strings for the available space.

    version 3.20, 01/02/2018
    build 600
  • Version 3.20 offers a fully configurable Results panel with panel rules. The old rather complicated display procedure for showing the panel results under all circumstances is now replaced by a large set of tests that is applied to every item on the Results panel rule list. All configuration issues are presented in a setup page. The standard setup resembles previous versions. This should accommodate for most wishes at local sites. There is a speed penalty when a division is calculated. This is now done separately for the results panel and for the graphical presentation. This takes some 65 ms extra.
  • The Results panel height is now adjusted automatically with a minimum of 5 lines and a maximum 13 lines.
  • All lengths and positions can be displayed in mm.
  • Improved failure reporting when loading reference files for 2D arrays.
  • Some subtle inconsistencies were found with the normalisation in the analysis engine and the transfer to the Results panel display procedure. This might give small differences with previous versions.
  • A new annotation:  (*) for applied normalisation data from the configuration data.
  • A preview of the keyboard buffer to avoid needless updates of the user interface.
  • FFF detection explicitely limited to photon data.
  • The measurement menu has been slightly reorganised. Less used items are moved to submenu's.
  • Introduction of Clear Merge source item in the Processing menu.
  • The 'Penumbra too wide message' is now limited to conventional profiles.
  • Minor bug fix: after using Merge an internal activity counter was not decreased, effectively disabling the acceptance of new data. Now a message is shown in such cases ('Analysis object reports busy state.').
  • Bug fix in shift procedure: When a profile is shifted (horizontally) and the normalisation position is set the the origin, the normalisation position should be kept at the origin and the normalisation value should be recalculated. This had only an effect on the Gamma calculation where dose differences are relevant. In all other there has been no effect at all of this bug.
    build 601, 10/02/2018
  • For mcc files crossline offset was linked to inline positions and vice versa in june 2016. This swap is reversed now.
  • Added a "trim" instruction when reading lines of a mcc file to remove illegal spaces in the data of home-made files.
    build 602, 14/02/2018
  • Minor display only bug fix: In the results panel lables that indicated a negative position (i.e. "X-12.0") were displayed with two negative signs: "X--12.0".
    build 603, 19/02/2018
  • The displayed text in the messages tab for electron pdd fits was different from the actually used model. It should be "pdd = I0/( 1 + I1*Exp((-mu1+(mu2+(mu3+mu4*(cm-d0))*(cm-d0))*(cm-d0))*(cm-d0)) + Ib*Exp(-mub*cm) ) + Ix*exp((-mx1+mx2*cm/100)*cm/100)".
  • For the pdd fits, the threads are now spread evenly over the available cpu's. Windows chooses other strategies by default.
    build 604, 22/02/2018
    In the Varian w2CAD format no field is defined for the Linac name. BistroMath therefore takes the comment field in a desparate attempt to get anything at all. In files generated by Eclipse the comment "(Open)" was found. Because of the brackets the complete string was discarded as Linac name. With no linac name at all comparison would fail with this file set as temporary reference (and Ignore Treatment Unit name checked). Only unchecking Check Temporary Scan Type would allow a comparison. Now only assumed illegal characters are stripped from the Linac name, in the example above leading to "Open". With this name the comparison is working again.
    build 605, 26/02/2018
    As correction on built 604 still the Linac name is clipped at the first space to be consistent with previous BistroMath versions. 
    build 606, 16/03/2018
  • When clicking on a Fit results table column, Bistromath returns to the primary Analysis tab.
  • Improved file reading for rfb files. Finally the bytes representing the number of files in a subgroup were identified. Previous versions of BistroMath mysteriously failed to find all scans in some of these files.
    build 607, 20/03/2018
  • Clarification for symmetry of FFF profiles: not the classic NCS symmetry definition but the Elevation (former: Linearity) is displayed in the standard rule set. In previous builds of version 3.20 this was hard coded. Now it is implemented as a rule in the standard rule set.
    build 608, 30/03/2018
  • In the division of two curves clipped data could lead to an invalid result and the whole curve would be discarded. Now the data range is limited to unclipped data only. 
    build 609, 03/04/2018
  • Diagonal detection is limited to fields larger than 2 cm as smaller fields widen because of penumbra effects. 
    build 610, 12/04/2018
  • Patch on build 600: the depth of a PDD-level xx, Dxx, was only available for predefined values (90%, 80%, 50% and 20%). Now a panel rule can be made for any level.

    version 3.09, 08/11/2017
    build 581
    Depending on the source format the diagonal scans could be presented as TA-GB instead of the intended GA-TB direction. In tracing back the origin of this phenomenon no errors were found in the interpretation file formats except for diagonals in the PTW StarCheck mcc format. In the central data preparation procedure the scan angle definition was erroneous due to the user mapping constraints. Note that the scan angle is always derived from the GTABUD axis conventions.
    For the StarCheck all data have been checked from and to the mcc format with absorbers put on selected positions.


    version 3.08, 30/07/2017
    build 574
  • The Mayneord transform was limited to SSD's larger then 50 cm in the code, making it non-functional for some orthovolt data. This limit is now set to 5 cm. When the Dmax for this transform was set negative or zero the actual Dmax of the measurement was taken. However, for orthovolt data a value of zero is meaningful here. Therefore now only a negative Dmax will be set to the actual Dmax.
  • The Mayneord item is moved from the Processing menu to the Measurement menu. As processing item it could be applied multiple times to one data set. In the new setup it can be included in the main analysis stream.
  • The Mayneord transform can now only be applied once to a measurement. Its application is now stored with the curve data as a separate flag.
  • The power function in the photon fit model is problematic for negative z values. In this case the power function is now replaced with the z value itself (effective setting the power value to 1).
  • The Mayneord transform state is now explicitely shown instead of the more generic "composite" state for the measurement in the Results panel.
  • The high order nature of the photon model leads to "resonances" of the model around the measured curve. Therefore the "error to noise ratio" (ENR) was introduced in version 2.86. For the orthovolt energies this leaves not enough weight to model the start of the curve. This can be overcome by switching off the ENR or giving the low z range a higher weight. The latter is now introduced. The user should experiment with optimal settings here.
  • A parameter limiting mechanism is added to the Nelder-Mead code. Now the parameter I1 is limited to a positive range.
  • The Nelder-Mead optimisation code was reviewed in a twofold manner. With some small reorganisations the code is made much faster without any functional changes. Afterwards this loop has been made thread-save and can now be started mutiple times in parallel on different cores. When all threads have completed, the best result is selected and redistributed on a possible restart of all threads. Therefore an evolution characteristic is now added to the original concept which improves the effectiveness significantly.
  • There is now an overview of all relevant parameters for the fit of PDD curves. The Settings tab and Advanced settings tab have been sligthly reorganised to accommodate the overview.
  • When there is still some analysis active, no new data are accepted to avoid multithreading conflicts.
  • Minor bug detected: with a multiple scan reference (i.e. a 2D array) the internal buffers were only partly cleared when the Settings tab was opened. On return to analys this made the reference effectively non-functional. This state could only be resolved by loading a scan with a single scan reference. Now these buffers are completely cleared, forcing the reload of a multiple scan reference.
    build 575, 04/08/2017
  • While testing the pdd fit on more data and different circumstances, it was observed that the initial data could produce an illegal starting point for the optimalisation. Now the initial data have been slightly changed. Also the assignment of the initial data is placed in a loop where, if needed, the initial values are changed with an increasing random fraction several times to find a legal starting point.
    build 576, 09/08/2017
  • The Mayneord transformation is now only applied if meaningful (the new SSD differs from the original) and when applied the reference pdd is reloaded again from either a now matching source or the temporary set reference.
    build 577, 12/09/2017
  • Stability precautions in the presentation layer by using the buffer only with valid (fit) data.
    build 578, 14/09/2017
  • In the Configuration tab the modality format (e.g. 'X10.0') is now set to two decimals to accommodate low energy x-rays like 'X0.05'. Note that all data found in a configuration file are now always forced to this new format.
    build 579, 28/09/2017
  • In the Files tab the data grid is made gray during file search and sorting as "busy" signal. Because the search process is handled in a separate thread the moment of finishing is unclear for the user otherwise. When finished it is set to the standard color again.
  • Minor bug fix: for diagonal scans the linac name and ennergy were lost in the reference file name; therefore references could not be found.
  • In the LoadReference routine a message is added that shows the standard name as derived from the measurment.


    version 3.07, 26/06/2017
    build 570
  • The pdd-fit is adapted for low and medium photon beams. The model itself is not changed but now the data up to the water surface are included when the found position of the dose maximum is smaller than 3 mm or available data stop withn 3 mm of the dose maximum. There were some stablity issues with the standard photon model for the low energy photon beams. Extra precautions have been taken to keep the model parameters within reasonable limits. Both the Reprocessing option (<Ctrl-P>) and Reset fit option (<Ctrl-Z>) can be used repeatedly to find better fits. Due to both the nature of the Nelder-Mead fit optimisation and the type of model, extreme values can occur. The Nelder-Mead code has been reviewed to handle multiple illegal error function outcomes in a better way.
  • Both the maximum of the Top model or the intersection of  the slopes (Slopes model) can be used to define the center of the field. Now this result can also be used as alignment criterium without choosing this as center definition.
  • At startup already available data on the clipboard are ignored.


    version 3.06, 25/05/2017
    build 567
  • Severe bug fix: Due to a procedure call with wrong default assumptions the results of the sigmoid calculation were (partially) plotted outside of the assigned memory borders. This may give unpredictable random errors. All users are therefore advised to update to this version. No more memory conflicts have been detected.
  • Major bug fix: the configuration tables in the Configuration Tab were messed up when edited. Effectively only direct editing of the configuration file  would set the correct information.
  • Major bug fix: editing the configuration tables in the Configuration Tab was nearly impossible as most non-numeric key input was intercepted by the menu system. Now this is disabled.
  • Bug fix: the activation of Generic Beam lists was not transferred from the user interface to the beam evaluation unit effectively making this feature only available if set in the configuration file. It now works as intended.
    build 568, 06/06/2017
  • Due to a mixup in build 567 the keyboard-only shortcuts were made available exlusively for the Configuration tab instead of all other tabs.
  • Bug fix: the mirror-function was reviewed for its effect with different profiles and centering options. It was concluded that the current practice, mirroring around the Center of Field (CoF) will suit for all scenarios. However the implementation effectively mirrored around half of the CoF value. This is corrected.
    build 569, 08/06/2017
  • For 2D-datasets the dataset was kept as reference after the mirror-function was stopped. Now all reference-buffers are fully cleared on unsetting this mode.
  • The SHA-256 checksum of BistroMath is now put in the Messages tab. This can be checked online, see for instance https://defuse.ca/checksums.htm#checksums.


    version 3.05, 13/04/2017
    build 564
  • Review of centering and normalisation choices in the Measurement menu: for FFF there can now be made a completely independent choice.
  • A more general approach for applying offsets in any direction to any profile is now offfered in the Settinsg tab. This can be practical for devices like 2D-arrays, which cannot set the origin easily. Note that these values are introduced in the preprocessing stage immediately after the axis mapping. The old PDD shift was set as a postprocessing shift.
  • FFF detection is limited to scans crossing the isocenter or pararallel to the main axes: the AB, GT and Angle type. A diagonal scan with a shifted measuring device will become a "free scan" when the origin is not crossed within 0.5 cm.
  • For SNC ICprofiler clipboard data additional frame data after the array data will be ignored. Multtiframe data (*.prm files) are not yet supported. This will be availabe in the near future.
    build 565
  • Bugfix: The appropriate choice for centering a FFF field was after application overwritten with the choice for a conventional field. All subsequent results were set with the "conventional" choice.
  • Minor bugfix: the FFF Center submenu in the Measurement menu would toggle the checked status each time the submenu was entered. This status is never used in the code but confusing for the user.
    build 566, 05/05/2017
  • Minor bugfix: in extreme examples found for the ICPROFILER the sigmoid calculation would fail. The validity of the result was not checked however in the results presentation, leading to a program error. This is now handled.


    version 3.04, 19/01/2017
    build 551
  • Several problems with failing sigmoid fits were found and resolved. One single unacceptable data point (position=0) would invalidate the whole fit. Also the acceptable average fit error was set to an absolute value, which would fail for data sets with very high values. Now this limit is made relative.
  • When the sigmoid model is used, the result for the unfileterd measurement are presented, because the model itself uses already a lot of points.
  • On initial preparation of read data the values ara rescaled when the maximum is larger than 1000 or smaller than 0.1.
  • The relative error is only displayed when then Calculated curve is also visible.
  • Merged pdd's ended up in the wrong (invisible) destination when they had different ssd's and were loaded in the "wrong" order. Now this ordering sensitivity is resolved.
  • For curves with multiple peaks (stripe like field setup) for levels above 65% the wrong position was found without local peak analysis.
  • FFF detection failed for the very steep penumbra's of the MRLinac.
  • The mirroring is now done around the chosen/calculated center of field.
  • The profile matching is improved to be stable for profiles without penumbra. A new inclusion criterium was introduced here.
  • Several minor issues were resolved.
    build 552, 24/01/2017
  • On export of MCC inplane profiles now always the TARGET_GUN direction is enforced. In previous versions the internal default GUN_TARGET was used, resulting in both axis direction reversal and sign inversion for the scan positions. Although correct, this deviates from the PTW standard.
  • A remaining issue with scan axis offset for MCC-export was resolved.
  • Resolved an issue with superfluous lines from the original file on overwriting an existing file.
    build 554, 05/02/2017
  • For FFF profiles the general choice for center of field to origin was ignored. Effectively the edge based center was set and (correctly) as such reported.
  • The option Detect FFF in the Measurement menu was not stored in any configuration or preset. This is now written as a preset (and therefore also in a configuration).
  • Also the optional choice to use the top of the FFF profile was ignored when the general choice for the center was set to origin. 
  • All items related to edge detection and sigmoid analysis are now part of a preset.
  • Unnessesary use of the sigmoid fit is avoided.
  • A textual clarification in the results panel: for FFF profiles there is, in general, not a meaningful 80-20 penumbra distance. As alternative the normalised slope of the sigmoid fit is presented.
    build 555, 09/02/2017
    Serious issues with matching introduced in build 551 were found. The problems originated from attempts to limit the search range for matching because a wide search range could lead to improper results. Now another strategy is followed.
  • When a profile was matched with itself the program would go into an infinite loop. This situation would occur when a temporary profile was loaded and matching was chosen as alignment method.
  • For FFF fields with missing penumbras the attempted strategy led to a far too large initial shift with improper results. Merging of such profiles with profile matching as merge parameter  would lead to garbish. 
    build 556, 12/02/2017
    Reviewed missing penumbra strategy for analysis of FFF profiles. In these cases the missing penubra is ignored. The derivative edge is set to the scan limit when this represents the highest found penumbra gradient. In this case the found edge position is assumed valid. The FFF slopes are always guaranteed to be centered around the top. 
    build 557, 15/02/2017
    Minor bugfix: when first a multiscan file from a 2D array was read including the according reference, then a watertank scan for the same machine would find the reference acceptable.
    build 558, 20/02/2017
    Bugfix: An extra test in the analysis introduced a division by zero error for pdd's.
    build 560, 21/03/2017
    Further input checking and range checking added. No functional changes.
    build 561, 27/03/2017

    version 3.03, 08/12/2016

The resultant virtual profile resembles pretty close a conventional profile. This is implemented here as a synthetic profile.
  • To make this work an ignorance mechanism was introduced to handle differences in parameters that usually should be equal for a comparison.
  • The coded names of the items in reference menu are changed. Please use the update option of the install program to make the ini files consistent. Note that this feature can be applied to any data directory.
  • A privacy statement has been added to the help file.
    build 535, 18/12/2016
  • The divide function offers a median filter as post-processing. This was and is intended for "shapeless" output. For synthetic (i.e. virtual) profiles. the post-processing is now switched off and a quadratic filter is applied instead.
  • A privacy statement and checksum are added to this help file. The privacy statement is also added to the install file.
    build 536, 20/12/2016
  • Systematic search for leftover possibilities for division by zero in the code.
    build 537, 22/12/2016
  • Minor bug fix: going to the Settings tab or Advanced settings tab would make the in-memory version of an automatically loaded reference invalid but still block reloading it again. No reference was used or displayed until another reference file was taken.
  • Clarification in the help file: the vertical range of the gamma plot is controlled by the Histogram range.  [based on user input, thank you!]

  • The histogram horizontal range and gamma plot vertical range increment steps are now controlled explicitely to produce nice steps. Previous versions could produce identical numbers on the scale because of the chosen number of digits in the lables.
    build 538, 27/12/2016
  • After detection of the first FFF profile, only once the help file is opened on the getting started with FFF topic. This action is stored in the used configuration file.
  • At last the context sensitive help for html-help type help files sorted out. Checked all pages for the correct context.

  • For reading the files in the Files tab now a separate analysis engine is used, avoiding any interference with possible data in shown in the Analysis tab.

    version 3.00, 13/10/2016
  • This version extends widely the possibilities for multiple references in a single file. Where in previous versions this feature was limited to files with a fixed structure, now any file can be used by creating an index to the file. Note that, by definition, these files cannot be loaded automatically. Also all supported binary files now are accepted as reference file. In previous versions only one singe scan from a binary file was taken. Please read the new overview topic.
  • All data file access was reviewed. Now always the data are placed in a memory stream immediately, which is more efficient for repeatedly opening the same data. In the alpha release one bug appeared here: when changing to another multi-reference file the stream was appended to the existing stream in stead of replaced.
  • Now any supported file including binary types can be used as reference. This opens the opportunity to open an unstructured rfb or mcc file as reference.
  • The alpha state of this version only is related to data reading and reference selection strategy. When new bugs are introduced they are only related to failed reading of data or not using the correct reference in exotic circumstances. The analysis as such has not been changed.
    build 517, 22/08/2016
  • PTW has updated the application BeamAdjust (we have now version 1.9.2.16). In previous versions the scan depth was set at a fixed value of 0.00 mm. In the new version you can set it to the actual depth. This might make old references invalid as they are still set to a depth of 0. The most sensible solution is to mass edit all reference and/or measurement files. I found fnr.exe (https://findandreplace.codeplex.com). This works straightforward. Search for mcc files (*.mcc) and do a find/replace for "SCAN_DEPTH=0.00" with "SCAN_DEPTH=50.00". There is no need to rename the reference files.
  • When an alias is added any existing index for the reference file will be rebuild.
  • File recognition for binary file types might fail in build 516 due to introduction of streams for thease file types. The file pointer was not set to the start on file load.
    build 519, 23/08/2016
  • Loading a binary file, which is not supported through the clipboard, will lock the clipboard; loading a text-based file will restore the last known state for the clipboard. The user can change the clipboard state at any time.
    build 523, 27/09/2016
  • Standard non-alpha release. One serious flaw was found. There was possible loss of reference files due to the usage of multi-scan references. The origin of this bug was the backward compatibility for the naming convention of reference files. Older files were renamed to the latest convention.
  • A number of tests for various combinations of measurements and refererence is described in the Comparison overview topic.
    build 525, 29/09/2016
  • Repeatedly reloading the same file failed to take the in-memory reference.
    build 526, 07/10/2016
  • Extra safety catches on disk file related operations.
    build 527, 13/10/2016
  • Merging of profiles with different setup parameters (i.e. scan direction, field size, depth, collimator) was and is not allowed. Now there is a message displayed in the status bar in these cases. [improvement based on user input, thank you!]
    build 528, 29/10/2016
