History v2.70 - v2.98


version 2.98, build 513, 26/07/2016
The introduction of FFF beams introduces new phenomena when changing beam parameters. In the discussions at my clinic this led to the conclusion that analysis based on the origin as beam center could be helpful. In the design of BistroMath the center was always defined as the avarage of left and right penumbra. 
  • In a new help topic all issues related to beam center and normalisation are presented. 
  • A new Center of field submenu in the Measurement menu offers three choices (penumbra, origin, maximum) for the beam center definition. The separate menu option Maximum is Center of Field therefore is removed.
  • The ordering of items in the Measurement menu is slightly changed for clarity.
  • Some <Shift+Alt> shortcut keys in the Measurement menu were changed to accomodate the new Center of field submenu.
  • Shifting to the origin is now based on the filtered version of the measurement, because that is also the base for presenting analysis results.
  • For FFF profiles the Top model curve is now plotted in a much higher resolution then the measurement step size.

    version 2.97, build 509, 07/07/2016
  • Minor bugfix: the Normalisation submenu did not respond.
  • A new FFF center submenu offers alternative methods to calculate the center of the field. New annotations have been assigned to the 'Center' value. After some discussion the slope area and central area are made configurable in the Settings tab.
  • A language thing: -ize words (except size) are replaced by -ise.

    version 2.96, build 506, 28/06/2016
    As general management feature to maintain a reliable setup now configuration file overloading is described in this help file. This has been designed as such from the very start of BistroMath. It comes to help with applying different tasks on top of a standard setup.
  • The limit for acceptable penumbra's is made accessible in the Advanced settings tab. The default is set to accept all penumbra's. The help file on this topic is clarified.
  • Changing tabs now clears the reference buffer to force a reload, except when it is a temporary reference.
  • The symmetry correction factor now clips when the profile falls below a certain fraction of the normalisation value.
  • In the Fit Results tab dmax is added. This value is not part of the model but used to rescale the model such that it produces 100 at dmax. The fit results are put on the clipboard immediately after the calculation, using the last known options and selection. 
    Review of the SpecialMode-functionality:
  • As addition to previous possibilities now any SpecialMode can be activated from the Processing menu. That state also can be saved to a configuration file.
  • SpecialMode2 is extended with reporting of the edge value, inflection point, user level position, symmetry, flatness, linac error, maximum, normalisation, division and gamma.
  • SpecialMode2 now also can be used for multiscan files.

    version 2.95, build 496, 04/06/2016
  • Bug report by user for reading MCC files: the values for SCAN_OFFAXIS_INPLANE and SCAN_OFFAXIS_CROSSPLANE were mixed up on conversion to the internal data format. A crossplane offset would shift the inplane scan positions.
  • Bug repair for the implementation of (switching off) axis preservation for exporting files of the Omnipro_v6 type in version 2.93. Actually this coding error failed to preserve the axis. Note that this bug also affects reference files written with version 2.93 and 2.94 when the T-direction is set to be negative. All other file types were implemented correctly.
  • The help file now contains a Quick Start guide. Work in progress. Comments welcomed.

    version 2.94, build 495, 23/05/2016
  • Bug report by user for reading and writing MCC files: new found values for the REF_SCAN_POSITIONS list were added to the already available values. Therefore this list would grow indefinitely. Because this list is not actively used in BistroMath but just stored for convenience when when writing MCC data, the same list would also grow in the written files. The effects, beside an increasing file length, for PTW software are unknown, but probably minor.

    version 2.93, build 494, 12/05/2016
  • Minor bug: when a generic data set representing a profile is forced to be used as pdd and the pdd fit is active and the mub_power is optimised, invalid fit data could lead to an unmanaged code failure.
  • Minor bug: the in-memory version of the reference was not not inverted when for the current measurement the axis setup was changed. Now the in-memory version is made invalid, forcing a reload of the reference.
  • Minor bug: after a code revision the linac name was not exported when writing Scanditronix files.
  • More explicit error messaging for wrong references. When the contents of a reference file are conflicting with its file name the reference is not used. This might be the case for inconsistent multi scan file references when the first scan header information is inconsistent with subsequent scan headers. Now, to surprise of the user, BistroMath will rightfully reject the reference for the second scan.
  • During the design of Bistromath a lot of effort was put in preserving data integrity on export. Therefore any axis swapping, which works on original data, was always compensated at export. However the user might choose to apply the axis swapping also to the exported data. Note that this may affect insitute wide data integrity when these data are mixed with original data.

    version 2.92, build 492, 14/04/2016
  • Minor bug fix: the Measurement option Generic to PDD did not work due to a code review before version 2.60. Not reported by any user.

    version 2.91, build 491, 31/03/2016
  • Patch for inconsistent output behavior of OmniPro v7.4 when the data in the "Normalised field" and "Current field" equal zero after making the data symmetric in OmniPro (message: "Maximum equals minimum").
  • Improved differentiation between true diagonal scans and "free scans". Diagonal scans should pass the origin and therefore the true field size can be evaluated from these scans. In previous versions the nearest measurement point should be within 0.5 cm of the origin. For the diagonals of the StarCheck Maxi the innermost points are at 1.43 cm. Now the distance between the scan line and the origin is evaluated.
  • Introduction of the Processing option Ignore name of Treatment Unit. By definition this option only works for temporary references. It is intended to compare identical beams of different linacs. The Aliases tab offers a more generalised approach which also will select references of a different linac.
  • In the Aliases tab now a message on available keys is displayed.
  • In the Files tab only files with a registered file extension are read, all other are ignored to avoid problems with strange formatted files. For .wda files the file name was not passed with the data, which only showed up as a problem in the Files tab. In multithreaded mode a message on completion is shown.

    version 2.90, build 486, 08/03/2016
  • Bug fix: The Measurement option Maximum is Center of Field was effectively ignored when activated. Now this calculation is executed elsewhere in the code.

    version 2.89, build 485, 29/02/2016
    Only changes to SpecialMode3:
  • Works now also for multiple-scan files.
  • The default output format can now be set at startup. Note that the last used output format is actually used.
  • Now the output can be either the original measurement or the filtered version of these data.

    version 2.88, build 484, 15/02/2016
    Display issues in the results panel solved:
  • The measurement date/time was accidently added to the d50 and d80 label for electron pdd's.
  • Any shift, preconfigured or manually applied, leads to an annotation for relevant labels.
  • When both absolute and relative flatness are given for a FFF-curve, the absolute flatness is followed by a "c". This actually stands for "%/cm".
    In some circumstances a message "Cannot focus on a disabled or invisible window" was displayed during startup. In the test situation this could only be forced by starting the application from a networked drive. The order of creation of both visible and invisible structures was carefully reviewed.
    For reading data from file, drag-and-drop and clipboard, the data are actually read to a stream and processed from that stream. The Editor is now filled after analysis from the same stream. Loading the editor is very time consuming for large files and can now be readily interrupted to load any other data.

    version 2.87, build 477, 02/02/2016
  • Minor bug: The state of the Detect FFF item in the Measurement menu could not be changed and was "on" after install. Not reported by anyone as a problem, but resolved now.
  • Logics of Setting/Unsetting a temporary reference have been changed from the old combination <Ctrl+T> as toggle function "set on/off" and <Ctrl+Shift+T> as "redefine" to the new combination <Ctrl+T> "set temporary reference" and <Ctrl+Shift+T> as "unset temporary reference".

    version 2.86, build 476, 12/01/2016
    While testing on other data the pdd model for photons showed to lead to a oscillating result. Despite recent code reviews the implementation of the Nelder-Mead optimisation worked flawless. It was the model itself that became unstable. This oscillation, induced by the buildup term, is now effectively dampened by introducing a buildup power factor in the photon model. The improvement is quit spectacular.
    After initial failing attempts to optimise also the buildup power factor now only the deviation from 1 (effectively no power factor) is included in the Nelder-Mead optimisation. When this still leads to unstable results the power factor can set fixed to any value between 1 and 2.  Also the Error to Noise Ratio is now added to the cost function. Both measures improve the outcome fully automatically in most cases. Alternatively some manual steering can be done.

    version 2.85, build 473, 04/01/2016
  • Testing on a predefined error value when this was set by the defined cost function attached to the Nelder-Mead module proved to be unreliable when comparing extreme values of floating point variables. This mechanism is needed to detect illegal combinations of parameters delivered to the cost function. With hindsight failing equality tests induced numerous extra runs of the procedure. In version 2.84 this was repaired by testing on reaching more then 90% of the extreme value. A small typing error was made in this process, leading to a suboptimal pdd fit. This is repaired here.
  • The space bar is used for quick selection of a graphical element (line or curve).
  • Resolved a graphical issue introduced in version 2.80. On changing the normalisation, the reference curve was updated only after reload of the same profile or loading another measurement.
  • For the linac symmetry error the sign of the error can now be set independently for GT and AB scans. For any other type of scan the sign is kept unchanged. The installation script will rename the item BeamRadiusSign to BeamRadiusSignGT in the configuration files. In institute.ini the item BeamRadiusSignAB is added.

    version 2.84, build 469, 20/12/2015
  • Spectacular processing speed improvement for large text-based data (files). This could take up to 5 seconds for a file of 1 MB and, with hindsight, was also noticable for small files. The bottleneck showed to be the transfer from the (rich text) editor to the (unformatted) text parsing module which is slow when the underlying data types are different. Now this transfer is done through an intermediate memory stream instead of direct copying.
  • Detection of FFF-profiles. Note that FFF-profiles have an alternative approach for analysis. This is denoted with a "F" for the flatness result. The absolute flatness is presented in percent/cm. The relative flatness remains unchanged and is still presented in percent.
  • The settings tab has slighty reorganised. A more fundamental approach to manage numerous parameters is prepared.
  • In the w2CAD format the original date string is now stored and copied to the appropriate field on import. Note that the comments field is used to transfer the linac name.
  • For recognised 2D-arrays the device name is taken as detector name.
  • Bug repair for recognised 2D-array data used as temporary reference. The current scan direction was used in all cases as reference. This bug was introduced in version 2.80 when the whole data input logistics were reviewed and temporary reference curves were kept in memory, thus avoiding storage to disk. Now not only the current scan is saved to memory but the complete file. From there the correct scan is loaded as needed. As added bonus still the correct scan direction is taken when the temporary file type check is switched off. Thus different linacs/beams might be compared.
  • The measurement menu has been reorganised and regrouped.
  • All lines in the graph area can be selected. When selected, the element is put to the foreground and the line width can be changed with the "{" and "}" key.

    version 2.83, build 462, 03/12/2015
  • Introduction of an additional sigmoid model based edge detection. This might improve sparse data sets. The related parameters are added to the Advanced settings tab.

    version 2.82, build 460, 23/10/2015
  • Minor bugfix: In SpecialMode2 file naming failed.
  • Support for multiple scan files in the (mass) file conversion tab.

    version 2.81, build 459, 05/10/2015
  • Bugfix: Due to a code reorganisation a temporary reference would not be shown or used in non standard installations. This is repaired.
  • Bugfix for OmniPro v7 data: when the maximum in ratio is low and a (therefore) a zero value was encountered, an inconsistent switch to another data column was made. Now the "Normalised Field" column is used in all these case. Note that the Normalised Field column is not corrected for the reference channel variations.
  • Tiny patch: quadratic filtering is applied after addition of two profiles. In a large code review for version 2.80 the data points from both profiles to be added were gathered using a the same routine that is used to evaluate the normalisation point. This makes the influence of both data sources on the result identical. However, the small calculation range used there may pass too much of the original noise. Therefore post-filtering is switched on again.

    version 2.80, build 456, 22/09/2015

    Introduction
    There have been numereous subversions, most of them in alpha state test mode. Some small issues have been solved afterwards. This build will be the final release of version 2.80.
    During testing in the production environment a new issue was found. One of the underlying problems showed to be the installation script which created the  institute.ini file in the (Windows7) path C:\ProgramData\. instead of the intended path C:\ProgramData\Bistromath where the application searches for it. All users are advised to run the install with this version. See also the topics on configuration and Installation issues.

    Overview
     New: For the MR-linac a dedicated water phantom with two detector positions is developed. The acquired data have a partial overlap. A full profile or depth dose curve has to be calculated from the combined signals. Matching is optionally used to stitch two partial profiles. To work succesfully with partial FFF-beams a number of revisions was needed.
     New: Mayneord correction for photon depth dose curves.
     New: Full support for the Sun Nuclear IC Profiler export format. This format can be exported from the Sun Nuclear software to file or clipboard. The latter provides a full link with BistroMath including dedicated reference files.
     New: Comparison of a profile with a mirrored copy and Maximum as center of field option as tools for FFF evaluation.
     New: Evaluation of position and level of maximum in depth dose curve based on a quadratic fit on top. This might give slight difference with previous versions.
    Improvements on the results panel including the composite attribute for curves.
    Improvements in matching for FFF and missing penumbra situations.
    Improvements and bug fixes for the PTW (Mephysto) .mcc format.
    Minor bug fixes.

    Details
  • Revision of the edge detection function. One missing penumbra would invalidate the calculated center of field and both edge positions. Now any succesfully found edge will be valid. Extra measures to exclude false edges are taken. See the detailed explanation at the Edge fallback topic.
  • Revision of match-function to handle also (FFF) beams with one missing penumbra. The matching area is now limited to provide roughly the same profile values at both sides of the area. An initial calculated optimal shift is used to set a starting value for the cost function. The revised edge detection was needed here to initialise the process on a single edge. The match cost function is now normalised on the number of succesfully matched data points, which avoids poor matches based on few points.
  • A minor revision of BistroMath's core function that evaluates the field borders of a profile. The results of a partial edge detection (see above) are now included in the final analysis results.
  • A small improvement of the BistroMath's core function for final analysis. The maximum of a depth dose curve is now fitted with a quadratic fit to evaluate the position and height of the maximum. This might lead to subtle differences with previous versions when the maximum is used for normalisation.
  • In early versions of BistroMath the original 3D coordinates of each data point were not stored in memory. On export the shift was applied to the reconstructed 3D coordinates. For these historical reasons the original 3D position data were not updated in previous versions when a shift was applied. This results in a wrong curve when the stored data of a merge (see below) were read as reference. Now all shift related data are updated. The Match function will become somewhat slower because it uses the Shift function frequently. For succesful merging (see below) the 3D-data are needed including any applied shifts. All shift-related code was removed from all export procedures.
  • Introduction of the composite attribute for curves. When set the presented text in the results panel is changed.
  • Alignment improved for partial curves: only the valid edges are used.
  • The division of two curves is done with filtered versions of these curves. The profiles were aligned again after this filtering including a possibly needed match which is time consuming. Now an already matched profile is not matched again.
  • All revisions above have been tested carefully and can handle all previously known non-standard situations as well as FFF beams with one missing penumbra. One original file of the .opab type with wedge profiles (toe at the target side, in IEC-601 conventions collimator=0°) was converted to the rfb format within OmniPro. This file was opened in BistroMath and showed the correct wedge orientation. This profile was exported from BistroMath as OmnPro-6-format, PTW .mcc, RFA-300 .asc, wms ascii .wtx and wms binary .wda. Al those files were loaded in BistroMath again succesfully with a correct axis representation.
  • As a result of all these revisions, for a FFF beam with one missing penumbra, the available field border will now be given. The field center and field width will still be unavailable.
  • Introduction of the Mayneord transform for photon depth dose curves. The Mayneord transform can be applied from the Processing menu. Use it with caution!
  • Introduction of the Merge function in the Processing menu. All backgrounds can be found in the Merging of scans topic.
  • The Merge function produces a composite curve which can be set as reference. The Merge function also works for partial depth dose curves measured at different SSD's. Dependent on the quality of the accompanying information with the PDD, extra settings may be needed.
  • The settings for Background subtraction, Resample grid, Pips pixel size and Histogram range are moved from the Settings tab to the Advanced Settings tab. In the Settings tab the Merge  and Mayneord settings are added.
  • The raw data of a reference are kept in memory for reuse. A temporary reference is also copied to this area now. In previous versions this was handled by writing a temporary data file. The new setup is much faster and more efficient. This avoids also the chance of any redundant file after exiting BistroMath.
  • Early file type recognition for files read through the clipboard/Editor reduces useless trials for different file types significantly.
  • The lower limit of the left axis in the graphics display is fixed to zero in the unzoomed state with a new option Left Axis lower limit is zero in the View menu.
  • Introduction of Mirrored measurement as Reference in the Processing menu. This is especially practical to evaluate symmetry in non-flat profiles like FFF beams.
  • In the results panel now any resampling is denoted with (r).
  • Smart cycling through multiple scan files. The <Home> jumps to the first valid measurement in the file. When there is more than one scan in a file the scan number is displayed with the bracketed original file format, for example "[omnipro_rfb 1/56]".
  • Introduction of Maximum as Center Position in the Measurement menu.
  • The keyboard shortcuts for the Normalisation sub menu have been changed to a <Ctrl>+<Shift> value.
  • For both matching and merging a faster two point interpolation is done instead of a quadratic fit on at least three points. This has speed advantages and also improves the tracability of results.
  • Support for the Sun Nuclear export file type of the ICprofiler. The device is added to the "multiple scan reference list" at installation.
  • For diagonal scans the start and end point of the scan are now indicated with combinations of T or G  and A or B (e.g. "TA") instead of "L" and "R".
  • The minimum required dose level (as fraction of the maximum) for acceptable normalisation at the origin can now be specified.
  • The logging level can be set in the Advanced Settings tab. More information on finding a reference file is logged when set to 2 or with failed attempts at log level 1.
  • Bugfix in Quadfit fallback mechanism. This was triggered continuously due to change in data underlying types in a library. This could lead to unpredictable results.
  • Bug fix for loading presets: when not in Advanced mode, the preset needed to be loaded twice, to get the User Dose Level accepted. Now Advanced mode is switched on temporarily and restored to the original state afterwards. The default value for the user dose level in the dose calculation engine is set to 50% instead of the (intentional) illegal value of -100%.
  • Minor bug fix: a (in the header reported) zero field size would lead to a needless rejection of the profile.
  • Minor bug fix: The user option to swap axis signs could lead to wrong identification of the "positive" or "negative" diagonal. Any user swapping is now ignored during that evaluation.
  • Minor bug fix: the Move profile to Origin was based on provisional results only. The final center position was only evaluated after the shift and therefore was not exactly in the origin. Now the exact center position is also evaluated before the shift.
  • Bug fix for freescan type. The distance to the starting point was incorrectly defined. Probably never used.
  • Bug fixes for PTW .mcc files.
    Patch for MCC-files: versions with unreliable information on the scan angle ("SCAN_ANGLE=90.00") were found in the wild. A scan angle value of 90 is now ignored.
    Due to an interpretation error the reported "SCAN_OFFAXIS_CROSSPLANE" / "SCAN_OFFAXIS_INPLANE" was mistakenly added to the crossplane / inplane positions instead of the intended offset in the inplane / crossplane position.
    Although the correct information on diagonals is detected this did not lead to the needed change of identity to "diagonal scan".
    The correct identification of "positive" and "negative" diagonal now is made.
  • Bug fixes for exported PTW .mcc files.
    File date/time and measurement date/time were not set correctly in the exported data.
    Also here the same issue on "SCAN_OFFAXIS_CROSSPLANE" / "SCAN_OFFAXIS_INPLANE" was resolved.
    During export the correct curve type information was not preserved and defaulted to "PDD".
    Improved export of diagonals.

    version 2.79 29/07/2015
  • Sorting added to the Files tab. Also some measures were taken to make this feature completely thread-safe.
  • Zooming by creating a rectangle in the graphics screen now handles curves attached to the right axis in the same was as the keyboard/view menu Zoom function does.
  • The size of all texts except the menu is now enlarged automatically within some limits when the window is enlarged. This is only active for the Analysis tab where all text positions are calculated on a form resize. All other tabs have a static design and therefore need a also a fixed font size.
  • With a refinement of the general reading strategy the correct conclusions on the file type are retained when a (possibly multi scan) file does not contain enough data points in a scan. This avoids needless attempts to read with alternative file types. As added bonus, the last displayed status message becomes more meaningful. Now multi scan files which combine real scans with single dose points "scans" can be read succesfully by discarding scans with too few data points automatically. This is the case with PTW mcc files generated by the MultiCheck program of PTW. When generated by BeamAdjust these files start with four legal scans. For this reason an upper limit of 4 scans was hard coded in Bistromath for all Starcheck based mcc-files in previous versions. This limitation is now lifted.
  • In the results panel now any rescaling that changes the SDD is denoted with (z). For PDD's results are denoted with (f) when the fitted data are used.
  • Bugfix: The quadratic filter and median filter did not copy the input to output when no repeated filtering was allowed. This would lead to undefined results. However, effects of this to edge detection related behavior, were never observed in previous versions with the tested data sets. Now it was detected on a PTW scan with varying steps from 1 to 4 mm. When combined with the standard filter range of 6 mm this results in a very limited filtering range.
  • Patch for OmniPro v7 files which use the date format 'MMM-dd-(yy)yy' or 'dd-MMM-(yy)yy'.

    version 2.78 05/06/2015
  • Introduction of an option menu in the Files tab for opening a file or scan and deleting a file. Opening a file was already provided for through double-clicking.
  • Increased file reading speed in the Files tab by applying a separate, interruptable, reading thread.
  • Improved logics for using multiple-scan references. In previous versions the reference menu item Device Specific reference was only enabled when a relevant data set was loaded. When it not was checked, it could only be checked after loading a relevant file, missing the opportunity to handle this file as intended. Now this menu item is always enabled when at least one applicable device is found in the configuration.
  • Although the correct configuration for multiple-scan references is written in institute.ini at install time, this information was lost after rewriting this file. Now, any available setup read at startup, will be written again when a configuration is saved. It is now also described in the 2D arrays topic and the configurations files section.
  • In the results panel now non-standard normalisation is denoted with (n).

    version 2.77 09/05/2015
  • Introduction of Ionisation to Dose conversion for electron depth dose curves. See Measurement menu, scientific backgrounds and implementation. This menu item cannot be checked to activate it for all following data like other items in the Measurement menu. Its intended use is application by the user (<Shift-D>) after the date have been read. When the conversion is succesfull the menu item will be checked to reflect the current state. It is always reset for the next curve. The user is responsible for applying this function to the appropriate data. This function can be applied only once on a data set. Returning to the original data is possible by rereading the data (<Ctrl-L>). The old Dose conversion option is now renamed more correctly to OD to Dose conversion.
  • Refinements in user interface of Fit Results tab (adjustment column width, show label in messagebar as response to click-event).

    version 2.76 15/04/2015
  • Minor bugfix in edge detection fallback method with derivatives. As interpolation method the peak in the derivative function is fitted with a second order function. There was no check on fit results outside the fit range with noisy or non-penumbra data points. Now the calculated peak position is guaranteed to be within the range of the used data. When an invalid result is detected the outer points of the data are discarded until a valid result is found.
  • For relative curves, especially depth dose data, the position of the maximum may vary wildly. Therefore the found normalisation position may be pulled away from the preferred position during analysis. Both absolute and relative normalisation position are now afterwards copied from the measurement.

    version 2.75 24/03/2015
  • More flexibility and support for dates with only two digits for the year and the am/pm time format. Both may be system dependent and are critical for detection of the OmniPro v7 format.
  • A patch (with full backward compatibility) for systems with a comma as decimal point separator (which is not preferred anyway). This lead in all previous versions to a wrong interpretation of the modality data in the configuration tab. As consequence the calculated division of measurement and reference could be normalised to a wrong level. On rewriting any configuration file these data will be separated with a " | ". Note that mixed usage of a network installation is problematic for old configuration files, these should be rewritten.
  • Minor bugfix: the resampling grid size was stored in the configuration by both form and calcuation engine under different names, leading to a zero value at start.
  • Minor bugfix: when no modality data were available, the [WellhoferObject] section was not written to a configuration file as this was considered to be an invalid situation in the past. Now these data are stored. No complaints on this were ever reported. Writing a configuration file is displayed on the statusbar.
  • Added a Redefine Temporary reference to the Processing menu.
  • Label "Profile step" in Advanced Settings tab changed to "Shift step". The lower boundary is set at 0.01 cm.
  • Automatic unzoom for pdd's made optional in View menu.
  • A line with meaningful data is added to the Configuration Tab when no data are found.

    version 2.74 04/02/2015
  • Minor bugfix in the configuration tab: changing the selected row in edit mode would not update the changes correctly.
  • Minor bugfix: show right axis labels always when a pdd-fit is done.
  • Minor update: changing the normalisation now forces immediate analysis without need to reload the data.
  • Minor update: In the Files tab the selection filter is explicitely set to  "*.*".
  • Minor update: In the Fit Results tab a separate checkbox for header information is introduced. The labels checkbox does not include the labels anymore. Preferences for both checkboxes are now written to any config or preset file. 
  • Compiler fix pack included.

    version 2.73 21/11/2014
  • A minor bugfix in the rather obscure hdf-format: positive values of the trackstart value were set to zero, leading to a shift of all voxels. Negative values were handled correctly. As this format was only used for film data, all positions were relative anyway. To demonstrate the interpretation (see description of format) in the selftest file selftest05_hdf.txt, the center of all voxels is shifted 0.01 cm to the left and, to compnsate this, the track start and stop are shifted 0.01 cm to the right. Therefore the center of all voxels in track coordinates is unchanged.
  • Minor bugfix: the ignore fie type checking for temporary files functionality was lost in version 2.71. Reviewed and tested.
  • Minor bugfix: for depth doses calculated curves presenting a division the configuration "normalise at dose max" the maximum of the relative curve was used. Of course the maximum of depth dose data itself should be used here. This lead to an inconsistent graphical representation without further consequences.
  • When the file type checking against the temporary reference fails, a message is put in the statusbar.
  • When the calculated curve is a gamma analysis, this is explicitely displayed as axis title on the right axis.
  • When used, the right axis scale is colored to emphasise its relation with its related curve. In most cases this will be the calculated curve. The activation of the right axis now responds much more natuaral to the Zoom function.
  • A selftest for gamma analysis for a FFF beam is added.

    version 2.72 24/10/2014
  • Rescaling of a measurement at some SSD to default SSD.
  • Comparison of rescaled measurements with references at default SSD. Only a geometrical scaling is applied.
  • Inclusion of SSD in axis title options.
  • Selftest for SSD rescaling.
  • Validation of the PTW format for phantom rotation. No problems found.

However, the implication is that the SSD will be part of the equivalence criteria. Therefore the default SSD is introduced in the Settings tab.
  • Introduction of measurement value points curve. The Points display size can be set.
  • A start has been made with the documentation of the config files.
